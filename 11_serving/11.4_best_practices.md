## 11.4 生产部署最佳实践

将 LLM 从实验环境部署到生产环境需要关注延迟、吞吐量、成本和可靠性之间的平衡。

### 11.4.1 关键性能指标

生产环境中最重要的性能指标包括：

- **首词延迟**（Time to First Token，TTFT）：从收到请求到返回第一个生成词元的时间，直接影响用户感知的响应速度。通常要求 < 500ms
- **每词延迟**（Time Per Output Token，TPOT）：生成每个后续词元的平均时间，影响流式输出的流畅度。通常要求 < 50ms
- **吞吐量**（Throughput）：单位时间内处理的请求数或生成的词元数
- **请求延迟 P99**：99% 的请求在此时间内完成，衡量尾部延迟的可控性

### 11.4.2 流式输出

**流式输出**（Streaming）是提升用户体验的关键技术。不等整个回答生成完毕再返回，而是**边生成边返回**——用户看到文字逐步出现，感知延迟大幅降低。

实现上，推理引擎在每个解码步将新生成的词元通过 Server-Sent Events（SSE）或 WebSocket 实时推送到客户端。主流推理引擎（vLLM、TGI 等）均原生支持流式输出。

### 11.4.3 负载均衡与自动扩展

高可用的 LLM 服务需要：

- **水平扩展**：多个推理实例分担流量，每个实例运行独立的推理引擎
- **智能路由**：根据请求队列长度、GPU 利用率等指标将新请求路由到负载最轻的实例
- **自动扩缩容**：根据流量自动增减 GPU 实例数量，平衡性能和成本

Kubernetes + NVIDIA GPU Operator 是当前最成熟的容器化 LLM 部署方案。KServe 和 Ray Serve 等框架提供了更高层的抽象。

### 11.4.4 成本优化策略

LLM 推理的 GPU 成本是生产运营的主要支出。常见的优化策略包括：

- **模型量化**：INT8 或 INT4 量化可将模型大小减半甚至四分之一，同比例降低 GPU 需求
- **合适的模型选择**：并非所有任务都需要最大的模型，较小的专门模型在特定任务上可能以更低成本获得同等效果
- **预算感知的调度**：根据请求优先级分配 GPU 资源，低优先级请求在空闲时处理
- **缓存与即时重用**：对重复或相似的请求缓存结果，避免重复推理

### 11.4.5 安全与合规

生产 LLM 服务还需关注：

- **输入过滤**：检测和阻断有害的用户输入（提示注入攻击、越狱尝试等）
- **输出审核**：对模型输出进行安全检测，过滤有害、敏感或不准确的内容
- **数据隐私**：确保用户数据不被泄露或用于训练，符合 GDPR 等合规要求
- **速率限制**：防止滥用和 DDoS 攻击
